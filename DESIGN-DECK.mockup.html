The information architecture, in words

  1. Presence strip (top, permanent). The soul of the product gets the primary chrome: mic orb (attention mode = ring color; VAD pulse when a turn opens; click cycles attention, hold = push-to-talk), live caption area (listening state → your transcript → route chip), the one input (typing = speaking, routed by the same arbitration as voice), agent-speaking slot (who + sentence + stop), history toggle, interrupt, connection. The status bar and the bottom feed strip are gone — their contents live here and in the drawer.
  2. Rail (left): agents, then repos. Agents keep the one-selection model from the product pass (click = voice-active + their pages + pinned review). New: a speaking halo when their words are being voiced, and repos get a standing Review button — review is a place you go, not a thing you wait for.
  3. Work (center). Unchanged in role: tabstrip + pages (agent card, screen, diff, doc, terminal). New: the ⊕ review tab affordance, product empty states with working buttons, and the annotation composer styled as a first-class object.
  4. Review ledger (right dock). Findings and questions are one ledger (a question is just an annotation without a line). The chat tab dies; replies render under their cards. Export stays.
  5. History drawer (summoned). The full voice feed — you/route/say/queued — slides from the strip when asked. It is deliberately not dockable: the conversation is never the center.
The one-input story, precisely: there is exactly one free-text box in the app, and it means "say this". It goes through say_as_user → arbitration → the active agent, identically to voice. Questions to an agent about work are annotations, created in context (diff line, page) and answered in the ledger. Nothing else accepts prose.

Underlying-logic changes this needs (small, all protocol-first)

  • page.publish control command — human-initiated publish: {workspace, type: "diff", source: {branch|pr|staged}}, resolved daemon-side in the workspace root by the same resolver agents use. This is the §3.2 sentence ("humans can also add a diff… no agent attached") that was speced and never built. The bridge verb can't do it — it demands a session.
  • workspace.open control command — mint a workspace from a path on the daemon host (validated real directory). Completes agentless parity: first-run users can review a repo before any agent ever attaches, exactly like diff-annotate.
  • Enrich speech.started/finished with who (call name) and the text being voiced. Today they carry only source + turn_id — not enough to say who is saying what without fragile client-side correlation.
  • Client-only: resizable grid with persisted sizes; presence strip; history drawer; ledger merge; empty states. No other daemon changes.

What stays untouched

The store/bus seam, the pages model, findings/asks convergence, diff rendering + interdiff chips, xterm terminal pages, the agent card, worktree spawn — all keep their logic. This is a re-skin of chrome + three protocol additions, not a rewrite. The 316-test protocol surface stays the source of truth.

Build order (each slice ends at a click-through checkpoint with you)

U0 — protocol first: page.publish, workspace.open, speech payload enrichment — tests at the command seam before any pixel moves.

Everything the UI will do exists as a tested command first. (Also answers your CLI question — see the session notes.)

U1 — the presence strip + one input: orb, captions, speaking slot, history drawer; feed strip and status bar removed; chat tab removed, ledger merged.

Checkpoint: you talk, you see it; Freya talks, you see it. If this fails the touch-test, U2 doesn't start.

U2 — review as a place: picker (branch/PR/staged), repo Review buttons, ⊕ tab, empty states with working actions, agentless flow end-to-end.

Checkpoint: the diff-annotate parity test — pick a branch, see the diff, flag a line, no agent running.

U3 — product polish: resizable/persisted panels, sans/mono type split, focus states, reduced-motion, light-theme pass.

Checkpoint: the "would a GitHub rando get it" walk-through, together.

Open forks — your call before U1

  • Orb interaction: proposal says click cycles attention (muted → wake → always) and press-and-hold is PTT. Alternative: click = PTT toggle, attention only via a small menu. Which matches your hands?
  • Input placement: proposed in the top strip (typing = speaking lives with the captions). If your muscle memory demands bottom, it can sit as a single bottom bar instead — but only one of the two exists.
  • History drawer scope: proposed voice-only (transcripts/routes/says). Findings activity could interleave there too — or stay purely in the ledger. Lean: voice-only.
